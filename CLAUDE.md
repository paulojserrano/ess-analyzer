## Project Overview

Desktop application (Tkinter GUI + headless CLI) for analysing ASRS/ESS warehouse log data exported from the Hairobotics system. Loads one or more Excel files (one per operational day), runs a suite of Plotly analyses, and generates a self-contained HTML report with interactive charts.

## File Descriptions

**`app.py`** — `AnalyzerApp` class (Tkinter). Five GUI panels: file loader, station config treeview, analysis selector checkboxes, progress/log output, report opener buttons. Background thread (`_bg_run`) runs the full analysis pipeline. Headless entry point: `_run_headless(path)`. Analysis execution order is defined at the top of the file.

**`config.py`** — Brand colours (`INK`, `ACCENT`), `TYPE_PALETTE` (6 colours, cycled for zone types), `STAGE_PALETTE` (8 colours for lifecycle stages), `STAGE_LABEL_MAP` (Chinese → English stage names), `TOTAL_DURATION_COL` (standard task-duration column), `AMR_DELIVERY_TYPE_HINT`, `ANALYSIS_MODULES` (ordered registry of `(key, label)` tuples).

**`data_loader.py`** — Sheet detection by column-signature matching (`callback`, `station`, `lifecycle`). Auto-detects LABOR station layout (`_auto_station_config`), lifecycle stage columns (`_auto_stage_config`), AMR robot type (`_detect_amr_type`). `build_config()` returns the runtime `cfg` dict, merging auto-detection with user overrides from `asrs_config.json`. Also: `filter_to_peak_day()`, `detect_data_date()`, `load_user_config()`.

**`report_builder.py`** — `generate_html_report(outdir, days, summary_registry)`. Builds a single self-contained HTML file with Plotly.js from CDN. Multi-day runs get a tab bar (Summary + one tab per day). Each chart is a card with title, source, method metadata, and embedded Plotly div. Chart resizing is handled on tab switch via JS.

**`analyses/__init__.py`** — Documents the `ChartResult` schema. Every `run()` must return a list of dicts with keys: `id`, `title`, `figure`, `source`, `method`, `export_hint`.

**`analyses/cycle_time.py`** — Most complex module (6 charts). Requires the `lifecycle` sheet. Internal helpers prefixed with `_` handle each sub-chart. Uses Pearson + Spearman correlation for demand-vs-cycle scatter.

**`analyses/fleet_utilization.py`** — Uses vectorised O(n²) overlap detection for concurrent delivery counting. Falls back to a loop for n > 5k rows.

**`analyses/summary.py`** — Called with `all_days: list[dict]` (not the standard `run(data, cfg)` signature). Only invoked when ≥2 days are loaded.

## Development Rules

### Context Awareness — Which file to touch

| Task type | File |
|---|---|
| Brand colours, stage label map, module registry | `config.py` |
| New analysis module | `analyses/<name>.py` + register in `config.py` `ANALYSIS_MODULES` |
| Chart visual changes (layout, colours, annotations) | The specific `analyses/*.py` module |
| Report HTML/CSS structure | `report_builder.py` |
| Excel sheet detection or column mapping | `data_loader.py` |
| GUI panels, buttons, treeview, threading | `app.py` |
| Cross-day summary charts | `analyses/summary.py` |
| Regenerate AI context index | Run `python ai_codex.py` |

### Adding a New Analysis Module

1. Create `analyses/<name>.py` — export `run(data: dict, cfg: dict) -> list[dict]`
2. Every returned dict must include all six `ChartResult` keys
3. Register in `config.py` → `ANALYSIS_MODULES` list (controls display order and checkbox label)
4. Add to the execution order list in `app.py` (`_bg_run`)
5. If the module requires the `lifecycle` sheet, guard with `if 'lifecycle' not in data: return []`
6. Run `python ai_codex.py` to update `.ai-codex/analyses-index.md`

### Hard Rules

- **NEVER import an analysis module directly in `app.py`** — the pipeline uses `importlib` / the registry pattern. Follow the existing dispatch pattern.
- **NEVER mutate the `data` dict** inside an analysis module. It is shared across all analyses for a given day.
- **NEVER use `print()` for user-facing output in analysis modules** — emit via the logging queue in `app.py` or raise an exception.
- **ALL column name strings are Chinese** — copy exact strings from `.ai-codex/data-schema.md`. Do not guess romanised versions.
- **cfg dict keys** — use only the documented keys from `.ai-codex/data-schema.md`. Do not add ad-hoc keys to cfg inside an analysis module.
- **ChartResult `id` must be unique** across all modules — use `snake_case`, prefixed by module name (e.g. `throughput_total`, `cycle_histogram`).
- **Plotly figures must be created with `go.Figure()`** — do not use `px.*` unless you verify the output is a `go.Figure` instance (report_builder calls `fig.to_html(full_html=False)`).

### Execution & Efficiency

- Use `.ai-codex/analyses-index.md` to identify which module owns a chart ID before opening any file.
- Use `.ai-codex/data-schema.md` to look up column names and cfg keys — do not grep the source for Chinese strings.
- Analysis modules are independent — changes to one do not affect others.

## Analysis Pipeline

Execution order (defined in `app.py`):

```
throughput → dwell_time → switch_time → cycle_time → retrieval → fleet_utilization
```

`summary` runs after all day-level analyses if `len(days) >= 2`.

`cycle_time` and `retrieval` are automatically skipped if the `lifecycle` sheet is absent from the loaded data.

## Data Flow

```
Excel file(s)
    └─► data_loader.load_data()        # sheet detection → {callback, station, lifecycle}
    └─► data_loader.build_config()     # auto-detect + asrs_config.json merge → cfg dict
            │
            ▼
    app.py _bg_run() [background thread]
            │  per day:
            │  ├─ filter_to_peak_day()
            │  ├─ analyses/throughput.run(data, cfg)   → list[ChartResult]
            │  ├─ analyses/dwell_time.run(data, cfg)   → list[ChartResult]
            │  ├─ ... (in order)
            │  └─ collect all ChartResult dicts
            │
            ▼
    report_builder.generate_html_report(outdir, days, summary_registry)
            └─► asrs_analysis_output/{date}/asrs_analysis_report.html
```

---

## ChartResult Schema

Every analysis module's `run()` must return a list of dicts conforming to:

```python
{
    "id":          str,        # unique snake_case identifier, e.g. "throughput_total"
    "title":       str,        # human-readable chart title
    "figure":      go.Figure,  # Plotly figure — must support .to_html(full_html=False)
    "source":      str,        # sheet(s) used, e.g. "callback sheet"
    "method":      str,        # one-sentence computation description
    "export_hint": str,        # associated Excel filename, or "" if none
}
```

---

## User Config Override

Place `asrs_config.json` in the same directory as the `.xlsx` file:

```json
{
    "station_types":  { "LABOR-1": "Zone A" },
    "design_rates":   { "LABOR-1": 120 },
    "type_colors":    { "Zone A": "#ff6b6b" },
    "amr_type":       "K50"
}
```

`data_loader.load_user_config()` merges this over auto-detected values.
